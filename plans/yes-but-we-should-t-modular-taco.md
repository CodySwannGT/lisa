# Add GitHub Issues — both as PRD source AND as JIRA replacement

## Context

PRs #434 (Confluence) and #435 (Linear) added new **PRD sources** alongside Notion. JIRA stayed the only **ticket destination**. Both `intake/SKILL.md:50` and `plan/SKILL.md:30` already carry "Not yet implemented. Stop and report" stubs for GitHub Issues, and the architecture is described as supporting it.

This plan closes both gaps in a single PR:
- GitHub Issues becomes a **PRD source** (mirror of Linear/Confluence pattern)
- GitHub Issues becomes a **JIRA replacement** (a configurable second tracker destination)
- Per the user: **no MCP server** — all GitHub interaction goes through the `gh` CLI, which is already used in repo scripts (`scripts/github-status-check.sh`, `scripts/lisa-commit-and-pr-local.sh`, etc.)
- Per the user, hierarchy uses **GitHub native sub-issues** (closest 1:1 to JIRA Epic→Story→Sub-task), accessed via `gh api graphql`
- Per the user, tracker selection lives in **`.lisa.config.json`** under a new `tracker` key

Outcome: a project can set `"tracker": "github"` in its lisa config and have the entire intake/plan/implement/verify/evidence lifecycle land in GitHub Issues instead of JIRA, while still being able to ingest PRDs from Notion / Confluence / Linear / GitHub.

## Part A — Tracker abstraction layer (the new seam)

The existing 3 PRD-to-tracker skills (`notion-to-jira`, `confluence-to-jira`, `linear-to-jira`) hard-code JIRA. The 11 `jira-*` skills are likewise vendor-locked. We introduce a thin vendor-neutral shim layer that dispatches by config.

| `tracker-validate/SKILL.md` | `lisa:jira-validate-ticket` | Same dispatch pattern |
| `tracker-verify/SKILL.md` | `lisa:jira-verify` | Same |
| `tracker-read/SKILL.md` | `lisa:jira-read-ticket` | Same |
| `tracker-evidence/SKILL.md` | `lisa:jira-evidence` | Same |
| `tracker-sync/SKILL.md` | `lisa:jira-sync` | Same |
| `tracker-add-journey/SKILL.md` | `lisa:jira-add-journey` | Same |
| `tracker-journey/SKILL.md` | `lisa:jira-journey` | Same |
| `tracker-create/SKILL.md` | `lisa:jira-create` | Same |

`jira-source-artifacts` is already read-only and vendor-neutral — reused as-is.

### Files to rename (`plugins/src/base/skills/`)

The 3 PRD-to-tracker skills get renamed and updated to call the shim layer:

| Before | After | Why |
|---|---|---|
| `notion-to-jira/` | `notion-to-tracker/` | These were always wrappers around JIRA-write; now they call `tracker-write` |
| `confluence-to-jira/` | `confluence-to-tracker/` | Same |
| `linear-to-jira/` | `linear-to-tracker/` | Same |

Internal `/jira-*` references inside these three skills become `/tracker-*`.

### Config schema change

`.lisa.config.json` (and `.lisa.config.local.json`) gains:

```json
{
  "tracker": "jira" | "github",
  "github": {
    "org": "<org-or-user>",
    "repo": "<repo>"
  }
}
```

Default: `"jira"` for back-compat. When `"github"`, the `github.org`/`github.repo` fields are required and validated by `tracker-validate`.

### GitHub Issues data model (encoded in skill docs)

Because GitHub Issues lacks JIRA's structured fields, every JIRA concept maps to one of: native sub-issue, label, body markdown section, or linked PR/comment.

| JIRA concept | GitHub Issues equivalent |
|---|---|
| Epic / Story / Sub-task hierarchy | **GitHub native sub-issues** via `gh api graphql` (`addSubIssue`, `removeSubIssue`) |
| Status (Ready / In Progress / Code Review / On Dev) | Labels: `status:ready`, `status:in-progress`, `status:code-review`, `status:on-dev`, `status:done` |
| Description (Coding Assistant / Developer / Stakeholder sections) | Markdown headings in issue body |
| Acceptance Criteria (Gherkin) | `## Acceptance Criteria` markdown section with Gherkin code-fence |
| Validation Journey | `## Validation Journey` markdown section |
| Source artifacts (Figma, Lovable, Loom, etc.) | `## Source Artifacts` markdown section per `jira-source-artifacts` taxonomy |
| Epic parent | Native parent sub-issue link |
| Blocks / is blocked by / relates to / duplicates / clones | Quick-action references in body (`Blocks #123`, `Blocked by #456`, `Relates to #789`, `Duplicates #321`) — `github-read-issue` parses these |
| Remote links (PRs, Confluence, dashboards) | `## Links` markdown section + native `Resolves #PR` |
| Components / Fix Version | Labels (`component:auth`, `fix-version:2.7.0`) |
| Story points / priority | Labels (`points:5`, `priority:high`) |
| Worklog | Comments (no native time tracking) |
| `Status=Ready` build trigger (the human-flipped signal) | `status:ready` label flipped by human |

### Lifecycle parity

Build intake batch (jira-build-intake → github-build-intake): scan `is:open label:status:ready` → claim by relabel → run `tracker-agent` build flow → on completion, relabel to `status:on-dev` and post comment.

PRD intake batch (linear/confluence/notion-prd-intake → github-prd-intake): scan `is:open label:prd-ready` → relabel to `prd-in-review` → run dry-run → relabel to `prd-blocked` (with sentinel comment on the source issue) or `prd-ticketed` (with link to created tickets in the configured destination tracker — could itself be GitHub Issues).

### Self-host edge case (GitHub PRDs → GitHub destination)

When tracker = github AND PRD source = github, both reads and writes hit the same repo. `github-prd-intake` must:
1. Avoid scanning the destination's own `status:ready` label (use `prd-ready` for source, `status:ready` for destination)
2. Document the label namespace separation in SKILL.md
