## T1 design

**Amendment (lead-ruled, DSS-1 resolver)**: materialized-default done-map entries (value exactly equal to the tracker default table for that env) are stripped at read time BEFORE union/alias assembly — invisible to the alias, the done-without-branch error, and the ladder join. First implemented with after-union ordering in `a73b7b8ff`; corrected to before-union in `ab79980a1` after the builder's pin exposed that after-union ordering empty-laddered prod-keyed projects with materialized maps (engine would no-op forever). Custom-valued entries are unchanged: they join the union, can block the sole alias, and error when branchless. Probe (`branches.prod: main` + fully materialized github default map): before `{"rungs":[],"terminalOnly":false,"terminalEnv":"prod"}`; after `{"rungs":[{"env":"prod","branch":"main","doneStatus":"status:done"}],"terminalOnly":true,"terminalEnv":"prod"}`.

**1. Module layout** (flat files, kane-style; no directory, no barrel; each ≤300 lines). Pure core in `src/core`, all process/exec/fetch IO in `src/cli` (ui-github-repo-gh.ts precedent):
- `src/core/deploy-status-transition.ts` — pure planner: `planTransitions(input): TransitionPlan` over refs × ladder × fetched item states; owns at-or-beyond, backward guard, leaf/container classification, terminal-closure decision, unconfigured-env no-op. Imports only `deploy-status-sync.js` types. Zero IO.
- `src/cli/deploy-status-refs.ts` — ref extraction (git + gh behind injectable deps).
- `src/cli/deploy-status-adapter.ts` — `TrackerAdapter` interface, `TrackerItemState`, `createTrackerAdapter(tracker, config, deps)`, dry-run recording wrapper.
- `src/cli/deploy-status-adapter-github.ts` / `-linear.ts` / `-jira.ts` — one file per vendor.
- `src/cli/deploy-status-sync-cmd.ts` — `runDeployStatusSync(options): Promise<number>`.
Rationale: matches existing flat convention (kane-cli, ui-github-repo-gh); the pure/IO split puts every execFile/fetch on the injectable-test seam.

**2. CLI surface.** `lisa deploy-status-sync` registered via **gate-commands** (extend `GateCommandDependencies` + `GATE_COMMAND_NAMES` — host-project CI callers must skip the root npm update check, update-check-hook.ts:51; ProgramDependencies still carries the runner like every other cmd). Inputs: exactly one of `--environment <env>` | `--branch <branch>` (branch→env via ladder reverse map; if the branch maps to multiple rungs — e.g. all envs → main — error asking for `--environment`); range as `--range <base>..<head>` or `--before <sha> --after <sha>` (normalized to `base..head`); `--dry-run`; `--json`; `--config <path>` (default `.lisa.config.json` at repo root). Output: one human line per item (`promoted <ref> → <doneStatus>`, `skipped <ref> (at-or-beyond: <status>)`, `skipped <ref> (container, leaf-only-lifecycle)`, `no-op: ...`) + summary; `--json` emits the full plan/result object. **Exit codes: 0 for success INCLUDING explanatory no-ops and skips; 1 only for genuine errors** (config validation from `resolveDeployLadder`, git/gh/API failures, unknown `--environment`, any per-item transition failure — remaining items still processed, then exit 1). Rationale: the unconfigured-env AC calls it an explanatory *no-op*; a red deploy job would gate deploys on tracker vocabulary, inverting the dependency. Real failures must redden CI so DSS-4 callers notice.

**3. Ref extraction: TS reimplementation** (do NOT spawn the .mjs — it's a host-project copy-overwrite artifact, absent from dist and unimportable). Same semantics: commits = `git rev-list <base>..<head>`; per-commit `git log -1 --format=%B` piped to `git interpret-trailers --parse` via execFile (stdin input), line regex `/^Work-Item:\s*(.+?)\s*$/i` (mirrors parseTrailers:347-356). PR bodies: per unique commit `gh api repos/{owner}/{repo}/commits/{sha}/pulls` (associated-PRs endpoint — merged PRs, no search-index lag; PR numbers deduped before body scan); scan body lines matching `/^(Refs|Closes)\b[:\s]/i`, extract tokens `owner/repo#N` | `#N` | `KEY-N` and canonicalize with the canonicalizeRef grammar (:254-283 — bare `#N` binds to the configured repo; out-of-repo/out-of-project refs are skipped with a report line, not an error). Result = deduped union on canonical form (AC 1).

**4. Adapter interface** (in `deploy-status-adapter.ts`):
```ts
export interface TrackerItemState { ref; type?: string; openChildren: number; currentStatus?: string; closed: boolean; }
export interface TrackerAdapter {
  fetchItemState(ref): Promise<TrackerItemState>;
  transitionToDone(ref, doneStatus): Promise<void>;
  upsertManagedComment(ref, body): Promise<"created" | "updated" | "unchanged">;
  closeNatively(ref): Promise<void>;
}
```
Leaf detection is NOT an adapter method — the pure core derives it from `TrackerItemState` (assertLeaf semantics :435-451: normalized type `epic` OR `openChildren > 0` ⇒ container); adapters stay dumb fetch/mutate. **5. Idempotency.** Marker `[lisa-deploy-status-sync]` (bracket precedent `[lisa-pr-link]`:22, `[claude-build-intake]`). Granularity: **one managed comment per item, find-or-update across all env events** — body is a deterministic function of the item's env-event history (env, branch, doneStatus, range head SHA; **no timestamps**). Re-run of the same event recomputes a byte-identical body ⇒ `unchanged` ⇒ the edit API call is skipped entirely — AC 3's "no duplicate comment" holds at the API-write level, and one comment total across dev→staging→prod prevents timeline spam. "No comment on skip" (AC 2), precisely: when the planner's action for an item is `skip-at-or-beyond`, the engine performs **zero adapter write calls** for that item — no transition, no comment upsert, not even an `unchanged` probe write (fetch-only). At-or-beyond check: map `currentStatus` into the ladder's doneStatus list; index(current) ≥ index(target) ⇒ skip.

**6. Leaf/parent + rollup note.** Containers (per #4 classification) are never transitioned or closed. The rollup note goes to **stdout/report AND a managed find-or-update comment on the container** using the same `[lisa-deploy-status-sync]` marker with rollup text ("container skipped per leaf-only-lifecycle; leaf items advance individually"). Rationale: Validation Journey step 5 observes "Epic remains open **with a rollup note**" — output-only risks a verification miss; find-or-update with a deterministic body means N deploys produce exactly one comment (edited, mostly `unchanged`), which answers the comment-spam objection.

**7. Backward-movement guard.** Rung index of an item = position of `currentStatus` among the ladder's doneStatus values (config-sourced vocabulary only). **No recognized done status** (never transitioned — e.g. `status:in-progress`, or no status label at all): index −1, below rung 0 ⇒ promote. **Unrecognized terminal-ish state** (closed at a status outside config vocabulary): skip with an explanation naming the observed value and the config surface (`<tracker> done map`) — config-bound vocabulary rule forbids guessing. Closed item: at-or-beyond by definition ⇒ skip (production target included — already terminal).

**8. Dry-run.** Short-circuits **writes only**: reads (`fetchItemState`, git, gh PR lookups) run for real so the printed plan reflects true ladder positions; the adapter is wrapped in a recording decorator (`withDryRun(inner)`) that delegates reads, records and suppresses `transitionToDone`/`upsertManagedComment`/`closeNatively`. Same decorator type is the unit-test double — one seam, two uses. Zero API writes guaranteed by construction (journey step 2/3).
