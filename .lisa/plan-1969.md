## T4 mitigation (builder, from verifier findings)

**Commit `42795513d`** fix(workflows): route gated deploy jobs to `<env>-deploy` (mitigation A) — templates (expo + nestjs) + tests + docs (root + copy-overwrite) + manifest regen, one commit, `--check` exit 0.

**EQ-1 (live FAIL → fixed)**: journey step 4 (approval-gate parity) regressed — with the DSS-3 declaration, an approval-gated run prompted the operator TWICE. Root cause: GitHub environment approvals are **per-job**, and the deploy job named the same protected env as `release_approval`, so the gate fired at both jobs. Pre-change there was one deploy job carrying the env (the gate only), hence one prompt.

**EQ-2 (finding recorded)**: a gated run records **two** deployments — the gate job against `<env>` and the deploy job against its env. Design §4's "exactly one deployment per run" was an overclaim for the gated path (already reworded in T3-D2; now finalized in the docs to the two-deployment reality).

**Fix (design §4 mitigation A)**: expo + nestjs deploy env resolves to `<env>-deploy` on gated runs, bare `<env>` otherwise:
`${{ needs.determine_environment.outputs.require_approval == 'true' && format('{0}-deploy', needs.determine_environment.outputs.approval_environment) || needs.determine_environment.outputs.approval_environment }}`
Host projects leave `<env>-deploy` unprotected → the gate fires once at `release_approval`, deploy proceeds without a second prompt; the `-deploy` suffix keeps the dev/staging/production substring for GitHub-for-Jira classification. nestjs keeps the mapping form (name = gated expr, url unchanged).

**rails/harper unchanged (confirmed reasoning)**: neither has a `determine_environment` job or approval wiring (rails approval is a documented gap, T1 §2/§8), so `require_approval` is never true and no gating path exists to collide with — their branch ternary needs no suffix. No gating machinery added to them.

**DSS-4 nuance (auto-inactivate)**: the deploy job's `<env>-deploy` deployment on gated runs carries a distinct env name from `<env>` — DSS-4's status-sync classification must treat `<env>-deploy` as the same logical env as `<env>` (substring match on dev/staging/production), and be aware GitHub may auto-inactivate the prior `<env>` deployment independently of `<env>-deploy`. Recorded here so DSS-4 accounts for the paired-deployment shape rather than assuming one deployment per gated run.

| File | Job | Why this job only |

| release.yml / release-rails.yml / build.yml | **none** — spec ruling: release.yml jobs (version/github_release/sentry_release/…) are release bookkeeping; `release_approval` L325-331 already declares its env (byte-for-byte unchanged); release-rails.yml `release` = version bump/tag; build.yml `build` = EAS compile, not a deploy. Declaring env on any of these double-records deployments per run. |

### 2. Env-name expression per file
- **expo, nestjs**: `environment: ${{ needs.determine_environment.outputs.approval_environment }}` — reuses the existing friendly-name resolution (github.environments key → deploy.branches key → branch fallback, jq at expo L78-83); matches envs provisioned by lisa-github-environments.sh (branch-policy pinned) and gives GitHub-for-Jira recognizable dev/staging/production names. `determine_environment` is already in both jobs' `needs`. No-config fallback = branch name — same degradation the approval gate already has; docs note only, do NOT change the resolution script.
- **rails** (`deploy_rails`) and **harper** (`deploy`): `environment: ${{ github.ref_name == 'main' && 'production' || github.ref_name }}` — no determine_environment exists; this mirrors rails' own APP_STAGE logic (L62) with zero new jobs; rails staging-push → `staging`, harper main-push → `production`, honest under workflow_dispatch on other refs. Adding full jq resolution jobs would violate smallest-diff.
- **environment.url**: nestjs only — mapping form `environment: { name: <expr>, url: ${{ steps.deployment_outputs.outputs.environment_url }} }` (`steps` context is valid in environment.url). Others use scalar form.

### 3. CDK ruling: amend the AC (option a)
cdk deploy.yml (L24-92) has determine_environment + release call ONLY — no deploy job exists; cdk/create-only holds nothing else deployable (ci.yml, cdk.json, tsconfig/vitest configs). Declaring env on the `release` workflow-call job is syntactically impossible; on a release.yml job it is wrong semantics + double-gates. Ruling: CDK is covered by (i) a commented `deploy:` snippet appended to the cdk template showing downstream projects how to declare `environment: ${{ needs.determine_environment.outputs.approval_environment }}` on their project-owned deploy job, and (ii) a GITHUB_ACTIONS.md note. Record as spec amendment on #1969 (mirrors DSS-1 alias amendment); AC cdk|production row satisfied by snippet+docs, not a live job.

### 4. Double-approval hazard — EMPIRICAL, resolve at T4
When require_approval=true, `release_approval` and the deploy job now name the SAME protected env. GitHub approvals are per run+environment, but whether a job *reaching* the env after approval re-prompts is not asserted here. T4 must run a protected sandbox env and observe. Decision tree:
- One approval unblocks both → ship as designed (preferred).
- Re-prompts → mitigation A: deploy env resolves to a distinct name when gated, e.g. `${{ needs.determine_environment.outputs.require_approval == 'true' && format('{0}-deploy', needs.determine_environment.outputs.approval_environment) || needs.determine_environment.outputs.approval_environment }}` (suffix keeps Jira substring classification); or B: always-distinct `<env>-deploy` name. Pick A if triggered; document either way.
