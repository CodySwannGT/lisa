# Usage Accounting

Lisa attaches token and cost telemetry to the artifacts its lifecycle creates and updates. The goal is to make AI spend visible at the same level people manage work: PRDs, tickets, PRs/evidence, debriefs, and the parent initiatives they roll into.

This rule is vendor-neutral. It defines the ledger shape, cost-estimation rules, attachment points, and rollup semantics. The `usage-accounting` skill performs the mechanical update; lifecycle flows and writer skills invoke it rather than inventing their own usage format.

## Core Contract

Every managed lifecycle artifact SHOULD carry a `## Lisa Usage` ledger. When the host cannot safely edit the artifact body, attach the same ledger as a comment on that artifact. A comment is still attached to the artifact; do not create a separate report as the only record.

Artifacts covered by this rule:

- PRDs created or updated by Research, Project Ideation, PRD intake, and Plan.
- Work items created or updated by Plan and Implement: Epics, Stories, Tasks, Sub-tasks, Bugs, Spikes, and Improvements.
- Verification evidence posted by Verify / tracker-evidence.
- Debrief triage documents.
- Intake cycle summaries when a batch run processes one or more PRDs or work items.

The ledger is append-or-regenerate managed content. Humans may read it, but agents own it. Never hand-author usage totals without a source.

## No Fabricated Telemetry

Usage data has a source. If the runtime exposes exact token usage or exact cost, record it as `observed`. If exact usage is not available but Lisa can estimate tokens from a known transcript and named tokenizer, record it as `estimated`. If neither is available, record `unavailable` with token fields set to `null`; do not guess.

Allowed source values:

| Source | Meaning |
| --- | --- |
| `observed` | Exact token counts and/or cost reported by the model runtime, API response, CLI transcript, or billing export. |
| `estimated` | Token counts estimated from transcript text with a named tokenizer/model. Cost may be estimated only from configured pricing. |
| `unavailable` | The flow ran, but no trustworthy token data was available. Token and cost fields are `null`. |

Rules:

- Never infer token counts from elapsed time, number of messages, PR size, line count, or vibes.
- Never silently convert `estimated` into `observed`.
- Never combine entries from different sources without preserving each entry's source.
- Never block artifact creation just because usage is unavailable; attach an `unavailable` row so the absence is explicit.

## Cost Estimation

Cost is derived from tokens only when Lisa has a trustworthy price source. Do not hard-code model prices in Lisa templates or skills; vendor pricing changes over time.

Resolution order:

1. Runtime-provided cost from an observed usage record.
2. Project config price map from `.lisa.config.local.json` then `.lisa.config.json` under `usage.pricing.models`.
3. `null` cost with `pricing_status=missing`.

Config shape:

```json
{
  "usage": {
    "currency": "USD",
    "pricing": {
      "source": "https://vendor.example/pricing",
      "snapshot": "2026-05-25",
      "models": {
        "provider/model-name": {
          "input_per_million": 0,
          "cached_input_per_million": 0,
          "output_per_million": 0,
          "reasoning_per_million": 0
        }
      }
    }
  }
}
```

Cost formula:

```text
cost =
  (input_tokens / 1_000_000 * input_per_million) +
  (cached_input_tokens / 1_000_000 * cached_input_per_million) +
  (output_tokens / 1_000_000 * output_per_million) +
  (reasoning_tokens / 1_000_000 * reasoning_per_million)
```

If a provider bills reasoning tokens as output tokens and the config omits `reasoning_per_million`, use the output rate only when the pricing source explicitly says so; otherwise leave reasoning cost out and set `pricing_status=partial`.

## Ledger Format

Render a readable table plus machine-readable tokens. The table helps humans. The HTML comments are the source of truth for parsers and rollups.

```markdown
## Lisa Usage

_Generated by `lisa:usage-accounting`. Costs are estimates unless the source says observed._

| Flow | Run | Model | Source | Input | Cached Input | Output | Reasoning | Total | Est. Cost |
| --- | --- | --- | --- | ---: | ---: | ---: | ---: | ---: | ---: |
| research | 2026-05-25T19:00:00Z/codex-abc123 | openai/gpt-5 | observed | 12000 | 3000 | 2500 | 0 | 17500 | USD 0.05 |

<!-- lisa:usage-entry entry_id=research:2026-05-25T19:00:00Z/codex-abc123 artifact_ref=github:CodySwannGT/app#123 artifact_type=prd parent_ref= flow=research run_id=2026-05-25T19:00:00Z/codex-abc123 provider=openai model=gpt-5 source=observed input_tokens=12000 cached_input_tokens=3000 output_tokens=2500 reasoning_tokens=0 total_tokens=17500 estimated_cost=0.05 currency=USD pricing_status=priced pricing_ref=config:usage.pricing.models.openai/gpt-5 -->

### Rollup

| Scope | Direct Tokens | Child Tokens | Total Tokens | Est. Cost |
| --- | ---: | ---: | ---: | ---: |
| this artifact | 17500 | 0 | 17500 | USD 0.05 |

<!-- lisa:usage-rollup artifact_ref=github:CodySwannGT/app#123 direct_tokens=17500 child_tokens=0 total_tokens=17500 estimated_cost=0.05 currency=USD child_refs= -->
```

Machine tokens use these fields in fixed order:

- `entry_id` — stable idempotency key, usually `<flow>:<run_id>:<artifact_ref>`.
- `artifact_ref` — vendor-qualified artifact id: `notion:<page-id>`, `confluence:<page-id>`, `github:<org>/<repo>#<n>`, `jira:<KEY>`, `linear:<identifier>`, `file:<absolute-path>`, or `pr:<org>/<repo>#<n>`.
- `artifact_type` — `prd`, `epic`, `story`, `task`, `sub-task`, `bug`, `spike`, `improvement`, `pr`, `evidence`, `debrief`, or `intake-cycle`.
- `parent_ref` — direct parent artifact ref when known; empty for top-level artifacts.
- `flow` — `research`, `plan`, `implement`, `verify`, `debrief`, `intake`, `repair-intake`, `monitor`, or another explicit Lisa command slug.
- `run_id` — runtime session/run id when available, otherwise an ISO timestamp plus a short random or branch-safe suffix.
- `provider` / `model` — provider and model used for the entry; `unknown` when unavailable.
- `source` — `observed`, `estimated`, or `unavailable`.
- Token fields — integer counts or `null`.
- `estimated_cost` — decimal string or `null`.
- `currency` — currency code, default `USD` when cost is present.
- `pricing_status` — `observed`, `priced`, `partial`, `missing`, or `unavailable`.
- `pricing_ref` — runtime billing export id, config key, or `none`.

## Attachment Points

### Research

Before invoking `prd-source-write`, collect usage for the Research flow and include a `## Lisa Usage` section in the PRD body. If the runtime has no token data, include one `unavailable` Research entry. `prd-source-write` and the vendor PRD writers preserve and update that section.

### Plan

Plan records direct planning usage on the source PRD. Each created work item also carries a ledger entry showing it was created by the Plan run. If the runtime cannot attribute planning tokens to individual tickets, keep the token-bearing Plan entry on the PRD and put a `source=unavailable` creation entry on each ticket; do not split a plan-run token total across tickets unless the runtime produced per-ticket usage.

After tickets are created, `prd-backlink` refreshes both:

- `## Tickets` / generated-work links.
- The PRD `## Lisa Usage` rollup, using the generated child refs.

### Implement

Implement records direct build usage on the work item being implemented. If the work item is a container and implementation happens on child leaves, attach direct entries to the child leaves and roll them up to the container.

### Verify

Verify records usage in the evidence artifact/comment and on the originating work item. `tracker-evidence` must include the usage ledger in the posted evidence comment and invoke `usage-accounting` so the work-item ledger and ancestor rollups are refreshed.

### Intake

Intake records a cycle-level entry for the batch orchestration itself and rolls up the artifacts processed in that cycle. A clean empty queue reports no artifact rows; a non-empty cycle attaches the cycle entry to each processed PRD or work item and includes totals in the final cycle summary.

### Debrief

Debrief records direct mining/synthesis usage on the triage document and rolls up the work-item and PR usage it analyzed. The debrief document should show both initiative delivery cost and debrief-analysis cost as separate rows.

## Rollup Semantics

Rollups are derived, not hand-written. A parent artifact total is:

```text
parent_total = parent_direct_entries + unique descendant_entries
```

Use these child sources:

1. `lisa:gw` generated-work tokens from `prd-backlink`.
2. Native tracker hierarchy when available.
3. Explicit `parent_ref` fields in `lisa:usage-entry` tokens.

Deduplicate by `entry_id`. If the same child appears through a native hierarchy and a documented section, count it once. If the same ticket is shared across two PRDs, each PRD may include it in its own rollup, but a single PRD rollup must not count it twice.

Roll up recursively:

- PRD totals include direct Research and Plan usage plus all generated work-item descendants and their Implement / Verify usage.
- Epic totals include direct Plan usage plus descendant Story / Task / Sub-task / Bug / Improvement usage.
- Story totals include direct Plan usage plus descendant leaf usage.
- Leaf totals include direct Plan / Implement / Verify usage.
- Evidence totals include direct Verify usage and can be referenced by the work item; do not count an evidence entry twice if it is already attached to the work item.
